Coverage Analysis in the .NET and Nunit Environment

2010-12-27
  • Performance
    • Programming
      • Unit Test
        • Windows
  • Coverage
  • Ncover
  • Nunit
  • Unit Test

Author: Doug Fleischman, Eastman Kodak Company – 04/18/07

In the Managed World, a free tool called Ncover (from www.ncover.org) allows us to perform coverage analysis in a simple inexpensive manner. Please be aware that there is another tool also called Ncover (from sourceforge) that works in a very different way.

What is nice about Ncover is that you do not need to instrument the code, and you can run it on Release executables as long as you have the companion PDB files.

After installation, add the directory location where Ncover is installed to your System Path environment variable. Running Ncover is simple, just run it at the command line with Nunit as the parameter. (Note: If Nunit is not in your system path value, you will need to use the full path) For example, for Pelican I can do this:

When I hit return I will get the following messages:

![clip_image002[5]](http://kevinxi.files.wordpress.com/2010/12/clip_image00251.jpg)

Now Nunit will come up:

![clip_image002[7]](http://kevinxi.files.wordpress.com/2010/12/clip_image00271.jpg)

Now you can select the tests you want to run and the press Run:

![clip_image002[9]](http://kevinxi.files.wordpress.com/2010/12/clip_image00291.jpg)

After all the tests are run, you can dismiss Nunit. Ncover will produce a file called “Coverage.xml” and exit.

Another free tool, called NcoverExplorer is available to view the Coverage.xml file. To use NcoverExplorer simply choose File – Open and navigate to the Coverage.xml file that you wish to view. The tool will show a tree view of the software in the left-hand pane and a view of the source code in the lower right pane:

![clip_image002[11]](http://kevinxi.files.wordpress.com/2010/12/clip_image002111.jpg)

In the tree view, the color codes have the following meaning:

  • Gray text - the namespace/class/method has zero coverage. (Icon will be grayed also) Red text - the namespace/class/method has partial coverage. Blue text - the namespace/class/method coverage has partial coverage exceeding a satisfaction threshold (default 95%).* Black text - the namespace/class/method has 100% coverage.

Once you have a coverage file open, you may select File – Merge and open another coverage file. By doing this you can see the combined coverage results from two different sets of tests. For example, you can merge the eFinder Unit Test coverage with the Pelican Unit Test coverage to see how Pelican components are covered across all testing.

The satisfaction threshold may be changed in the View – Options window:

![clip_image002[13]](http://kevinxi.files.wordpress.com/2010/12/clip_image00213.jpg)

The source code highlighting works like so:

  • Blue background - this section of code was visited (a tooltip tells you how many times).* Red background - this section of code was not visited.

NcoverExplorer also permits you to create html report of the code coverage. If you select View – Reports and the select Module/Class Summary like so:

![clip_image002[15]](http://kevinxi.files.wordpress.com/2010/12/clip_image00215.jpg)

You will get a CoverageReport.html file that looks like this:

![clip_image002[17]](http://kevinxi.files.wordpress.com/2010/12/clip_image002171.jpg)

Running from the Command Line

Ncover and Nunit may be run from the command line, which is what would probably be done for post-build testing in the Build environment.

Here is an example of how to run Nunit from the command line:

nunit-console.exe /labels PelicanUnitTests.dll > UnitTestResults.txt

Again, if Nunit is not in your path, you will have to provide the complete path. The resulting test results file will look something like this:

![clip_image002[19]](http://kevinxi.files.wordpress.com/2010/12/clip_image002191.jpg)

If you want to combine this with coverage analysis, you could do something like the following. (Note: the command would be all on one line)

Ncover.console.exe //x PelicanCoverage.xml nunit-console.exe /labels PelicanUnitTests.dll > UnitTestResults.txt

The //x option specifies that we want to store the coverage data in the file “PelicanCoverage.xml”.

The file “PelicanCoverage.xml” can then be browsed using NcoverExplorer, however the source files must be in sync with the coverage file in order for this to be effective.

NcoverExplorer may be run from the command line to generate a coverage report. (Command would be all on one line)

NcoverExplorer.console.exe /html:PelicanCoverage.html /report:4 PelicanCoverage.xml

The /html option means that we want to generate a coverage report in HTML format and call it PelicanCoverage.html. The option /report:4 means that we would like to generate a Module and Class summary. The final argument is the Ncover-generated coverage file that we would like to parse.
